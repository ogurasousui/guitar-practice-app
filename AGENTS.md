# AGENTS.md

このリポジトリは、ギター練習用Webアプリ `Guitar Practice` の開発リポジトリです。

GitHub Pagesで公開できる静的Webアプリとして実装します。MVPでは `Vite + TypeScript + React` を使い、ドラムとベースのバッキングは Web Audio API で生成します。

## 開発フロー

- `main` は公開用ブランチとする。
- 開発は必ず作業ブランチで行う。
- 変更はPR経由で `main` に入れる。
- `main` にマージされた変更はGitHub Pagesへ自動公開される前提で扱う。

## ブランチ名

ブランチ名は以下の形式にする。

- 機能追加: `feature/<内容>`
- 修正: `fix/<内容>`
- ドキュメント: `docs/<内容>`

例:

- `feature/audio-engine`
- `fix/bpm-slider`
- `docs/development-flow`

## コミットメッセージ

コミットメッセージは Conventional Commits 風にする。

例:

- `feat: add audio engine`
- `fix: correct bpm slider range`
- `docs: add development flow`
- `chore: update dependencies`

## PRルール

- 1つの目的につき1PRにする。
- 目的が混ざる変更は分ける。
- PRには必ず以下を書く。
  - 概要
  - 確認方法
  - 影響範囲
- 大きめの作業だけIssueを作る。
- 小さい修正や軽い改善はPRだけでよい。
- Issue紐付けはマージ条件にしない。

## マージ条件

- `npm run build` が通っていること。
- テストやlintを追加した後は、必要に応じて必須条件へ追加する。
- CodexレビューでP0/P1相当の指摘がある場合、未解決のままマージしない。
- Codexレビューの指摘が誤検知だと判断した場合は、PRコメントに理由を書けばマージしてよい。
- Codexレビュー自体が完了しない場合は、人間判断で進めてよい。

## UI変更時の確認基準

UIを変更した場合は、以下を確認する。

- PC幅で表示崩れがないこと。
- スマホ幅で表示崩れがないこと。
- スクリーンショット添付は必須ではない。
- 大きなUI変更ではスクリーンショット添付を推奨する。

## 音に関わる変更の確認基準

音源、テンポ、再生制御に関わる変更では、以下を確認する。

- 再生できること。
- 停止できること。
- BPM変更でテンポが変わること。
- 実際に音を聴く確認は推奨だが必須ではない。
- 複数ブラウザ確認は大きな変更時のみ推奨する。

## Codex作業ルール

Codexに実装依頼があった場合は、原則として以下まで行う。

- 調査する。
- 実装する。
- `npm run build` で確認する。
- 確認結果を報告する。
- 必要ならPR作成まで進める。

Codexが変更してよい範囲は、依頼内容に必要な範囲だけとする。

- 不要なリファクタはしない。
- ついで修正は別PRに分ける。
- 目的が混ざる変更は避ける。

## Review guidelines

Codexレビューでは、アプリが壊れるリスクを最優先で確認する。

P0/P1相当の重大な問題だけコメントする。細かい好み、将来改善、軽微な文言調整は原則コメントしない。

特に以下を重大問題として扱う。

- `npm run build` が通らない変更。
- GitHub Pages公開を壊す変更。
- 再生、停止、BPM変更が壊れる変更。
- PC幅またはスマホ幅での表示崩れ。
- 依頼範囲外の大きな変更。

## GitHub上のCodex運用

- PR作成時にCodexの自動レビューを行う。
- GitHub側でCodex Cloud設定とAutomatic reviewsを有効化する。
- 必要に応じて手動で `@codex review` を使ってよい。
- Codexレビューの指摘を修正する場合は、人間が妥当と判断した指摘だけ `@codex fix ...` で依頼する。
- すべての指摘を自動修正させない。
- Codexによる修正後も `npm run build` の確認を必須にする。
